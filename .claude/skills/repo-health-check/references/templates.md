## SUMMARY.md Template

```markdown
# Health Report: {owner}/{repo}

> Generated: {YYYY-MM-DD}
> Visibility: {Public|Private}
> Health Score: {score}/{max} ({percentage}%)

## Score Breakdown

| Tier | Earned | Max | Bar |
|------|--------|-----|-----|
| Tier 1 — Required | {n} | {max} | {bar} |
| Tier 2 — Recommended | {n} | {max} | {bar} |
| Tier 3 — Nice to Have | {n} | {max} | {bar} |

## Action Items

| # | Item | Tier | Points | Status |
|---|------|------|--------|--------|
| 1 | [README](tier-1--readme.md) | 1 | 4 | FAIL |
| 2 | ... | | | |

## Passing Checks

- **LICENSE** — MIT License (1,080 bytes)
- ...
```

The progress bar uses `█` for filled and `░` for empty, 8 characters wide.

## Action Item Template

Each `[FAIL]` or `[WARN]` item gets its own file:

```markdown
# {Check Name}

| Field | Value |
|-------|-------|
| **Repository** | `{owner}/{repo}` |
| **Tier** | {1|2|3} — {Required|Recommended|Nice to Have} |
| **Points** | {points} |
| **Status** | {FAIL|WARN} |
| **Detected** | {YYYY-MM-DD} |

## What's Missing

{One sentence: what the check found.}

## Why It Matters

{Two to three sentences explaining the impact. Tailor to the repo context — e.g., mention "private repo" or "fork" if relevant.}

## How to Fix

### Quick Fix

```bash
{single command using the actual owner/repo}
```

### Full Solution

{Longer explanation with file templates if needed.}

## Acceptance Criteria

- [ ] {Concrete, verifiable condition matching the check's pass condition}
- [ ] {Second condition if needed}

## References

- {Link to relevant GitHub docs}
```

### Template Guidelines

- **What's Missing**: Keep it factual, one sentence. "No README.md file exists." not "You should have a README."
- **Why It Matters**: Explain the *impact*, not just restate the check. "Without a README, visitors have no context..." not "A README is required."
- **How to Fix**: The quick fix should be a single copy-pasteable command. The full solution provides context for more thoughtful implementation.
- **Acceptance Criteria**: Map directly to the check's pass condition. Use checkboxes so the user can track completion.

## Suggestions per Check

When filling out the "How to Fix" section, here are recommended fixes for each check. Adapt these to the specific repo — use the actual `{owner}/{repo}`, detected tech stack, license type, etc.

### README (Tier 1)
- Quick: `echo "# {repo}\n\nDescription" > README.md`
- Full: Provide a template with Getting Started, Usage, and License sections
- Acceptance: README.md exists and is > 500 bytes

### LICENSE (Tier 1)
- Quick: `curl -o LICENSE https://choosealicense.com/licenses/mit/` (adjust to detected or suggested license)
- Acceptance: LICENSE file exists with a recognized license

### Description (Tier 1)
- Quick: `gh repo edit {owner}/{repo} --description "your description"`
- Acceptance: Non-empty description string

### Branch Protection (Tier 1)
- Quick: `gh api repos/{owner}/{repo}/branches/{branch}/protection -X PUT --input <json>`
- Full: Include require reviews, enforce admins, require status checks
- Note: If default branch name is unusual, suggest renaming to `main`
- Acceptance: Protection API returns 200

### .gitignore (Tier 2)
- Quick: `curl -o .gitignore https://raw.githubusercontent.com/github/gitignore/main/{Template}.gitignore`
- Choose template based on detected tech stack (VisualStudio.gitignore for .NET, Node.gitignore for JS, etc.)
- Acceptance: .gitignore exists

### CI/CD Workflows (Tier 2)
- Provide a starter workflow matching the tech stack
- For .NET: `dotnet build` + `dotnet test`
- For Node: `npm ci` + `npm test`
- Acceptance: At least one .yml/.yaml in .github/workflows/

### CODEOWNERS (Tier 2)
- Quick: `echo "* @{owner}" > .github/CODEOWNERS`
- Acceptance: CODEOWNERS in any standard location

### Issue Templates (Tier 2)
- Provide bug_report.md and feature_request.md starters
- Acceptance: .github/ISSUE_TEMPLATE/ with at least one file

### PR Template (Tier 2)
- Provide a template with Summary, Related Issue, Changes, Checklist
- Acceptance: PR template in any standard location

### Topics (Tier 2)
- Quick: `gh repo edit {owner}/{repo} --add-topic {topic1} --add-topic {topic2}`
- Suggest topics based on detected tech stack and repo purpose
- Acceptance: At least one topic set

### SECURITY.md (Tier 3)
- Provide a template with vulnerability reporting instructions
- Acceptance: SECURITY.md exists

### CONTRIBUTING.md (Tier 3)
- Provide a template with setup, style guide, and PR process
- Acceptance: CONTRIBUTING.md exists

### Security Alerts (Tier 3)
- Quick: `gh api repos/{owner}/{repo}/vulnerability-alerts -X PUT`
- Full: Also enable Dependabot security updates and optionally add dependabot.yml
- Acceptance: Vulnerability alerts enabled, no open critical/high alerts
