# Backlog Dashboard

Scan all backlog items, display a cross-repo dashboard with scores and progress, and recommend what to work on next.

## Prerequisites

- Backlog items must exist under `backlog/` in the working directory (generated by `repo-health-check`)

## Input

No input required. The skill scans all `backlog/` subdirectories automatically.

Optional filters the user might provide:
- A specific repo: "show backlog for phmatray/NewSLN"
- A specific tier: "show tier 1 items"
- Only failures: "show remaining failures"

## Step 1 — Discover Repositories

Scan `backlog/` for all `{owner}_{repo}/` directories. Each directory may contain one or more report folders (e.g., `health-report/`, `health-report-2026-02-26/`).

For each report folder, read `SUMMARY.md` to extract:
- Health score (earned / max / percentage)
- Tier breakdown (earned per tier)
- Generated date
- Visibility (Public / Private)

If multiple report folders exist for the same repo, use the most recent one (by date in the folder name, or the undated `health-report/` as the baseline).

## Step 2 — Collect Item Status

For each report, scan all `tier-*` markdown files. Parse each file's metadata table to extract:
- Check name (from the `# Title`)
- Tier and points
- Status (FAIL, PASS, WARN)

Count totals per repo:
- Items total, passed, failed, warned
- Points earned vs possible

## Step 3 — Display the Dashboard

### Multi-repo overview (when multiple repos exist)

```
## Backlog Dashboard

| Repository | Score | Progress | Tier 1 | Tier 2 | Tier 3 | Last Audit |
|------------|-------|----------|--------|--------|--------|------------|
| phmatray/NewSLN | 10/31 (32%) | ██░░░░░░ | 2/4 | 1/6 | 0/3 | 2026-02-26 |
| phmatray/OtherRepo | 24/28 (86%) | ██████░░ | 4/4 | 4/4 | 0/2 | 2026-02-20 |

Total items: 25 | Passed: 14 | Remaining: 11
```

The Progress column uses `█` and `░` (8 chars wide) based on the score percentage. The Tier columns show `passed/total` item counts (not points).

Order remaining items by tier (1 first), then by points (highest first).

## Step 4 — Recommend Next Action

After the dashboard, suggest the highest-impact next action:

```
### Recommended Next

The highest-impact item is **README** (Tier 1, 4 points).
To apply it:

  /apply-backlog-item backlog/phmatray_NewSLN/health-report/tier-1--readme.md
```

Selection logic:
1. Pick the repo with the lowest health score percentage
2. Within that repo, pick the lowest tier number (Tier 1 > Tier 2 > Tier 3)
3. Within that tier, pick the highest point value
4. If there's a tie, pick the first one alphabetically

### Stale report detection

If the audit date is more than 30 days old, add a note:

```
> This report is 45 days old. Consider re-running the audit:
>   /repo-health-check {owner}/{repo}
```

## Step 5 — Quick-apply prompt

If the user says something like "apply it" or "fix it" after seeing the recommendation, treat that as a trigger for the `apply-backlog-item` skill with the recommended item's path.

## Edge Cases

- **No backlog directory**: Tell the user no audits have been run yet and suggest: `/repo-health-check {owner}/{repo}`
- **All items PASS**: Display a congratulatory message with the final score
- **WARN items**: Show them separately — they aren't actionable without permission changes
- **Multiple reports per repo**: Show the most recent. Mention older reports exist if the user might want to compare
