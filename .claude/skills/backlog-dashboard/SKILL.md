# Backlog Dashboard

Scan all backlog items (health findings and issues), display a cross-repo dashboard with scores and progress, and recommend what to work on next.

## Prerequisites

- Backlog items must exist under `backlog/` in the working directory (generated by `repo-scan`)

## Input

No input required. The skill scans all `backlog/` subdirectories automatically.

Optional filters the user might provide:
- A specific repo: "show backlog for phmatray/NewSLN"
- A specific source: "show health items" or "show issues only"
- A specific tier: "show tier 1 items"
- Only failures: "show remaining failures"

## Step 1 — Discover Repositories

Scan `backlog/` for all `{owner}_{repo}/` directories. Each directory should contain:
- `SUMMARY.md` — the unified repo summary
- `health/` — health check findings (optional, may not exist if all checks pass)
- `issues/` — open GitHub issues (optional, may not exist if no issues)

For each repo, read `SUMMARY.md` to extract:
- Health score (earned / max / percentage)
- Tier breakdown (earned per tier)
- Open issue count
- Generated date
- Visibility (Public / Private)

### Legacy format support

If a repo has the old `health-report/` directory structure (from the previous `repo-health-check` skill) instead of the new `health/` + `issues/` structure, still read it. Parse the old `health-report/SUMMARY.md` the same way. Note it as "legacy format" in the display so the user knows to re-scan.

## Step 2 — Collect Item Status

For each repo, scan both `health/` and `issues/` directories.

### Health items (health/tier-*.md)

Parse each file's metadata table to extract:
- Check name (from the `# Title`)
- Tier and points
- Status (FAIL, PASS, WARN)

Count totals per repo:
- Items total, passed, failed, warned
- Points earned vs possible

### Issue items (issues/issue-*.md)

Parse each file's metadata table to extract:
- Issue number and title
- Labels
- Status (OPEN, PR CREATED, CLOSED)

Count totals per repo:
- Issues total, open, with PRs, closed

## Step 4 — Recommend Next Action

After the dashboard, suggest the highest-impact next action:

```
### Recommended Next

The highest-impact item is **README** (Health — Tier 1, 4 points).
To apply it:

  /apply-backlog-item backlog/phmatray_NewSLN/health/tier-1--readme.md
```

Selection logic:
1. Pick the repo with the lowest health score percentage
2. Health items take priority over issues (they improve the repo's foundation)
3. Within health items, pick the lowest tier number (Tier 1 > Tier 2 > Tier 3)
4. Within that tier, pick the highest point value
5. If all health items are done, recommend the oldest open issue
6. If there's a tie, pick the first one alphabetically

### Stale scan detection

If the scan date is more than 30 days old, add a note:

```
> This scan is 45 days old. Consider re-running:
>   /repo-scan {owner}/{repo}
```

## Step 5 — Quick-apply prompt

If the user says something like "apply it" or "fix it" after seeing the recommendation, treat that as a trigger for the `apply-backlog-item` skill with the recommended item's path.

## Edge Cases

- **No backlog directory**: Tell the user no scans have been run yet and suggest: `/repo-scan {owner}/{repo}`
- **All items done**: Display a congratulatory message with the final score
- **WARN items**: Show them separately — they aren't actionable without permission changes
- **Legacy format repos**: Show them with a "(legacy — re-scan recommended)" note
- **Multiple report folders per repo**: If a repo has both `health-report/` (old) and `health/` (new), prefer the new format. Mention the old one exists if the user might want to compare.
